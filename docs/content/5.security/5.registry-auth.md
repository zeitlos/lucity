---
title: "Registry Authentication"
description: "How container image registry credentials are scoped and managed"
---

Lucity uses Zot as its OCI registry for user workload images. Registry authentication determines who can push and pull images, and scoped credentials ensure one workspace can't tamper with another's images.

## Current Model

The platform currently uses a shared registry credential for all build operations. The credential is created as a Kubernetes Secret from Helm values and mounted in two places:

- **BuildKit daemon**: for pushing built images to the registry
- **Build Job pods**: for registry authentication during the build process

Both Secrets are generated by the same Helm template, so they stay in sync on every deploy. The build namespace Secret is not manually mirrored; Helm creates it alongside the platform namespace Secret.

## How Auth Works

Registry authentication uses Docker's standard `config.json` format:

```json
{
  "auths": {
    "registry.example.com:5000": {
      "auth": "<base64(username:password)>"
    }
  }
}
```

This is mounted at `/etc/registry-auth/` and referenced via the `DOCKER_CONFIG` environment variable. Both BuildKit and the build Job pods use this standard mechanism. No proprietary auth flow, no platform-specific token format.

## Workspace-Scoped Credentials (Planned)

The shared credential model works for single-tenant deployments, but in a multi-workspace setup, a compromised build in workspace A could theoretically overwrite images belonging to workspace B.

Zot supports path-based access control with glob patterns:

```json
{
  "accessControl": {
    "repositories": {
      "workspace-a/**": {
        "policies": [
          { "users": ["ws-a-builder"], "actions": ["read", "create", "update"] }
        ]
      },
      "workspace-b/**": {
        "policies": [
          { "users": ["ws-b-builder"], "actions": ["read", "create", "update"] }
        ]
      }
    }
  }
}
```

Each workspace gets its own htpasswd user, restricted to pushing images under its own path. Workspace A's credentials can't read or overwrite workspace B's images.

### Implementation Options

**htpasswd per workspace** (simpler): when a workspace is created, generate an htpasswd entry and update the Zot config. Static configuration, requires Zot config reload on workspace changes. Good for small-to-medium number of workspaces.

**Bearer token auth via OIDC** (dynamic): Zot supports Docker v2 bearer token authentication. A token server issues short-lived JWTs with repository-scoped `access` claims. No config reload needed when workspaces change. More infrastructure to build and maintain, but scales to many workspaces.

The Docker v2 token spec expects an `access` claim in the JWT:

```json
{
  "access": [
    {
      "type": "repository",
      "name": "workspace-a/api",
      "actions": ["push", "pull"]
    }
  ]
}
```

Logto (the platform's identity provider) supports custom token claims via JavaScript functions that run on token issuance. This could potentially mint Docker v2-compatible tokens directly, eliminating the need for a separate token service. This integration is under evaluation.

## Image Integrity

Every built image is tagged with the git commit SHA and labeled with OCI-standard metadata:

```
org.opencontainers.image.source: "https://github.com/acme/myapp"
org.opencontainers.image.revision: "a1b2c3d"
lucity.dev/built-by: "lucity-builder"
lucity.dev/service: "api"
```

These labels provide traceability from running container back to exact source commit. Combined with scoped credentials, this ensures that the image running in production was built from the code you expect, by the platform you trust, and pushed by credentials that belong to your workspace.
