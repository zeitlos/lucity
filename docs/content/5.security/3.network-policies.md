---
title: "Network Policies"
description: "How Lucity restricts network traffic between workloads and platform components"
---

Every namespace managed by Lucity starts with a deny-all NetworkPolicy. Nothing talks to anything unless explicitly allowed. This is the opposite of Kubernetes' default behavior, where all pods can reach all other pods across all namespaces.

## Build Namespace (`lucity-builds`)

Build pods have the most restrictive network policy. They can reach exactly four things:

| Destination | Port | Why |
|-------------|------|-----|
| CoreDNS (`kube-system`) | 53 UDP/TCP | DNS resolution |
| BuildKit (`lucity-system`) | 1234 TCP | Send build plans |
| OCI Registry (`lucity-system`) | 5000 TCP | (Reserved for direct push if needed) |
| Public internet | 80, 443 TCP | Fetch dependencies (npm, pip, go modules) |

Everything else is blocked. The cluster-internal CIDRs `10.244.0.0/16` (pod network) and `10.96.0.0/12` (service network) are explicitly denied for external egress, so build pods can't probe other services, the Kubernetes API, or workloads in other namespaces.

## Platform Namespace (`lucity-system`)

Platform services have component-specific policies:

**BuildKit** accepts ingress only from:
- Build Job pods in `lucity-builds` (labeled `lucity.dev/component: build`)
- The builder service in `lucity-system`

**Platform services** (gateway, builder, deployer, packager) communicate freely within `lucity-system` via gRPC. The gateway is the only service exposed to external traffic via HTTPRoute.

## User Workload Namespaces

Each project environment gets its own namespace (e.g., `myapp-production`). Workload network policies are defined in the `lucity-app` Helm chart:

- Services within the same namespace can communicate freely (same project, same environment)
- Cross-namespace traffic is blocked by default
- Public services are exposed via Gateway API HTTPRoutes, not by opening NetworkPolicy holes

## What the Policies Don't Cover

NetworkPolicies operate at L3/L4 (IP addresses and ports). They don't inspect HTTP headers, authenticate requests, or enforce application-level authorization. If two pods are allowed to communicate by NetworkPolicy, they can send any traffic on the allowed port.

For application-level security (authentication, authorization, rate limiting), rely on your application code and the platform's OIDC integration. A service mesh would add L7 policies but also adds significant operational complexity. For most deployments, NetworkPolicies plus application-level auth is the right trade-off.

## Verifying Policies

You can verify NetworkPolicies are enforced by checking from inside a build pod:

```bash
# This should timeout (blocked by NetworkPolicy)
curl -m 5 https://kubernetes.default.svc/version

# This should also timeout (cross-namespace blocked)
curl -m 5 http://lucity-gateway.lucity-system:8080/health
```

If either of these succeeds, the NetworkPolicy is not applied. Check that your CNI plugin supports NetworkPolicies (Cilium, Calico, and most modern CNIs do; Flannel does not).
