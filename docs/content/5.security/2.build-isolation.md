---
title: "Build Isolation"
description: "How Lucity isolates build workloads from platform components"
---

Builds are the riskiest operation a PaaS performs. You're taking someone's source code, running detection heuristics over it, and executing arbitrary build commands to produce a container image. Lucity separates every part of this pipeline from the platform itself.

## Architecture

The build pipeline has two components:

| Component | Namespace | What it does |
|-----------|-----------|-------------|
| **Build Job** | `lucity-builds` | Clones the repo, runs railpack to detect the framework, sends the build plan to BuildKit |
| **BuildKit** | `lucity-system` | Executes the actual build instructions (`RUN` commands), produces and pushes the OCI image |

The Build Job never executes user code. It reads files (package.json, go.mod, requirements.txt) to detect the language and framework, generates a build plan, and sends it to BuildKit as an LLB graph over gRPC. The Job is an orchestrator, not an executor.

BuildKit is where user-provided build commands actually run. BuildKit's OCI worker isolates each build step in its own container namespace. A `RUN npm install` in your build doesn't execute on the host or inside the BuildKit daemon process; it runs in a sandboxed execution environment managed by BuildKit.

## Why a Separate Namespace

Build Jobs run in `lucity-builds`, not `lucity-system`. This matters because:

- **DNS isolation**: Build pods can't resolve `lucity-gateway`, `lucity-deployer`, or other platform service names. Their DNS search domain is `lucity-builds.svc.cluster.local`, not `lucity-system.svc.cluster.local`.
- **Blast radius**: If railpack or the git clone process has a vulnerability that's exploited via a malicious repository, the attacker lands in an empty namespace with no platform components.
- **Network policy boundary**: The `lucity-builds` namespace has its own deny-all NetworkPolicy with selective egress rules. Build pods can only reach BuildKit, the OCI registry, and the public internet (for fetching dependencies).
- **RBAC isolation**: The `lucity-build-runner` ServiceAccount can only patch its own Job (to write build results as annotations). It has no access to Deployments, Secrets, ConfigMaps, or anything else.

## Kubernetes API Access

Build pods cannot reach the Kubernetes API server. The NetworkPolicy blocks all traffic to the service CIDR (`10.96.0.0/12`), which includes `kubernetes.default.svc` at `10.96.0.1`. The `/version` endpoint, which responds without authentication on most clusters, is not reachable from build pods.

## Build Job Security Context

Every build Job pod runs with a hardened security context:

```yaml
securityContext:
  runAsNonRoot: true
  seccompProfile:
    type: RuntimeDefault
containers:
  - securityContext:
      allowPrivilegeEscalation: false
      capabilities:
        drop: ["ALL"]
```

No root, no extra capabilities, no privilege escalation. The seccomp profile blocks dangerous syscalls like `mount`, `ptrace`, and `reboot` at the kernel level.

## Build Job Lifecycle

1. Gateway calls `StartBuild()` on the builder service
2. Builder creates a Kubernetes Job in `lucity-builds`
3. The Job pod clones the repo, runs railpack detection, sends LLB to BuildKit
4. BuildKit builds the image and pushes it to the OCI registry
5. The Job pod annotates itself with the image reference and digest
6. Builder polls the Job status every 3 seconds until completion
7. Jobs auto-delete after 24 hours via TTL

Build results (image reference, digest, or error message) are stored as annotations on the Job object. The builder service reads these annotations when the Job completes. No shared filesystem, no database, no message queue.

## BuildKit Considerations

BuildKit runs in `lucity-system` because it's a long-lived platform component with persistent cache storage and registry credentials. It requires `seccompProfile: Unconfined` for OCI image building (a requirement of BuildKit's OCI worker, not a Lucity choice).

BuildKit's own NetworkPolicy restricts ingress to only the builder service and build Job pods (identified by the `lucity.dev/component: build` label from the `lucity-builds` namespace). No other pods can connect to BuildKit.

The registry authentication secret (Docker config.json) is mounted on the BuildKit daemon, not passed into the build execution environment. A malicious `RUN` command cannot read BuildKit's filesystem mounts.
