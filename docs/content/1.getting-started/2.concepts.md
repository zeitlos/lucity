Before you start building, it helps to know how Lucity thinks about your infrastructure. These aren't proprietary abstractions. They map directly to Kubernetes and GitOps primitives. We just made them easier to work with.

## Projects

A project maps to a GitHub repository. It's the top-level container for everything related to your application: services, environments, and deployments all live under a project.

When you create a project, Lucity connects to your source repo on GitHub. This connection is **read-only**. The platform never writes a commit, a file, or a webhook config to your repo. Your source code is sacred.

Behind the scenes, Lucity also creates a **GitOps repository** to manage your deployment configuration. This is where Helm values, environment overrides, and deployment state live. You own both repos, but Lucity only touches the GitOps one.

## Services

A service is a deployable unit within your project. Think of it as a single process: a web server, an API, a background worker.

Lucity auto-detects services from your source code using [railpack](https://github.com/nichochar/railpack). It figures out the language, framework, build command, and start command without you writing a Dockerfile. If railpack gets it wrong (rare, but it happens), you can always configure a service manually with a specific port and public/private setting.

A project can have multiple services. A common pattern is an API service and a frontend service living in the same repo.

## Environments

Environments are isolated deployment targets. Each one gets its own Kubernetes namespace, its own set of Helm values, and its own running workloads.

By default, Lucity creates three permanent environments: **development**, **staging**, and **production**. These stick around for the lifetime of the project.

Then there are **ephemeral environments**, PR preview environments that are created automatically when you open a pull request and cleaned up when the PR is merged or closed. Your reviewers can click a link and see exactly what the code change looks like running on real infrastructure. No more "I'll test it after merge." Test it now.

## Builds

When you deploy, Lucity builds a container image from your source code. Railpack handles the detection and build plan, so no Dockerfile is needed (though you can bring your own if you prefer).

Builds move through a clear set of phases:

1. **QUEUED**: the build is waiting to start
2. **CLONING**: pulling your source code from GitHub
3. **BUILDING**: compiling, bundling, doing the actual work
4. **PUSHING**: uploading the finished image to the OCI registry (Zot)
5. **SUCCEEDED**: done, the image is ready for deployment

If something goes wrong, the build fails with logs you can actually read. No 500-line stack traces from a mystery build system.

## Deployments

After a build succeeds, the new image tag is committed to your GitOps repository and ArgoCD syncs it to Kubernetes. That's it. A Git commit triggers a deployment, and ArgoCD makes the cluster match what's in the repo.

You can watch deployments happen in real time with streaming logs in the dashboard. When ArgoCD reports the rollout is healthy, you'll see it. When something goes wrong, you'll see that too, with the actual error, not a generic "deployment failed" message.

## Promotion

Here's where it gets good. Instead of rebuilding your app for each environment (and praying the staging build is identical to what you tested), you **promote** an image from one environment to another.

The image that passed your staging tests is the exact same image that runs in production. Same bytes, same digest, same everything. Promotion just copies the image tag between environment configs — no rebuild, no re-download, no surprises.

This is how you stop saying "it worked in staging" and start knowing it works in production.

## Ejectability

This is the big one. The thing no other PaaS gives you.

At any point, you can run `lucity eject` and get a complete, self-contained export of your entire setup:

- **Helm charts**: the full `lucity-app` chart with all templates
- **ArgoCD Application manifests**: ready to apply
- **Environment overrides**: your dev/staging/prod values files
- **A setup guide**: prerequisites, commands, how to modify things

The output has **zero Lucity dependencies**. Point your own ArgoCD at the exported repo and run independently. You're not renting your infrastructure from us — you own it, and you can take it with you whenever you want.

Try asking Railway or Heroku for that. (Spoiler: they'll offer you a CSV export and a pat on the back.)

## Two-Repository Model

Lucity maintains two repositories per project:

1. **Your source repo** on GitHub. This is yours. Lucity reads from it but never, ever writes to it. Not a commit, not a file, not even a `.lucity.yaml` config file. Your repo stays clean.

2. **The GitOps repo**, managed entirely by the platform. This contains Helm values for each environment, organized in a clean directory structure. Every deployment is a Git commit here, giving you a full audit trail and the ability to roll back by reverting a commit.

This separation keeps your application code and deployment configuration independent. You can change your deployment strategy without touching your source code, and vice versa.

---

These concepts map directly to standard Kubernetes and GitOps primitives. There's no proprietary magic — just orchestration on top of tools you already know. If you ever want to peek behind the curtain, `kubectl get namespaces -l lucity.dev/project` will show you exactly what Lucity created.
