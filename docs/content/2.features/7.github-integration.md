Lucity integrates with GitHub for source code access, authentication, and automated deployments. Your GitHub account is your identity, your repositories are your projects, and webhooks tie it all together.

## Sign in with GitHub

Authentication is OAuth-based. Sign in with your GitHub account and you're in — no separate credentials to manage, no invitation emails, no password reset flows. Your GitHub identity (login, name, avatar) carries over to the Lucity dashboard.

Since Lucity is single-tenant (one instance per organization), access control is straightforward. Everyone on the instance is part of the same team.

## Connect a repository

When creating a project, browse your accessible GitHub repositories and pick one. Lucity uses the GitHub API to read your source code for builds and service detection.

The keyword here is **read**. Lucity needs to see your code to build it. It never needs to write to your repo. No generated config files get committed, no CI pipeline YAML appears in a pull request you didn't expect, no mysterious bot commits in your history.

## Automate with webhooks

Configure webhooks on your GitHub repo to enable automatic deployments:

- **Push to main**: triggers a build and deploy to your development environment. Push code, get a running deployment. The feedback loop that makes everything else faster.
- **Pull request opened**: creates an ephemeral preview environment, builds the PR branch, and deploys it.
- **Push to PR branch**: rebuilds and redeploys the preview environment with the latest changes.
- **Pull request merged or closed**: cleans up the ephemeral environment. No orphaned namespaces, no forgotten resources.

The webhook service receives these events and routes them to the appropriate services. It's the glue between GitHub's event system and Lucity's deployment pipeline.

## Get PR preview environments

This is where webhooks really shine. Open a pull request and Lucity:

1. Creates an ephemeral environment (its own namespace, its own config).
2. Builds the PR branch.
3. Deploys it to the preview environment.
4. Updates on every push to the branch.
5. Cleans up when the PR is merged or closed.

Your reviewers get a live, running version of the changes without anyone manually deploying to a shared staging environment and yelling "don't deploy to staging, I'm testing something" in Slack.

Ephemeral environments are lightweight: minimal resource allocations, same chart, same deployment pipeline. They're real environments, not simulated previews.

## Trust that your repo stays sacred

This bears repeating: Lucity reads from your GitHub repository but never commits to it. No `.lucity.yml` in your root directory. No generated Dockerfiles. No workflow files injected into `.github/`. Nothing.

All platform-managed configuration lives in the GitOps repository, which Lucity creates and manages separately. Your source repo is yours, exactly as you left it.
